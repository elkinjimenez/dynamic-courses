¿Qué es RabbitMQ?

Historia, propósito y cuándo elegirlo sobre otras soluciones

RabbitMQ es un message broker open-source: un servidor intermediario que recibe mensajes de unas aplicaciones y los entrega a otras. Está escrito en Erlang, fue creado en 2007 y hoy es uno de los más usados en el mundo. Implementa el protocolo AMQP 0-9-1, que define exactamente cómo viajan los mensajes entre productores, el broker y consumidores.

La idea central es simple: en vez de que el Servicio A llame directamente al Servicio B y espere su respuesta (y falle si B está caído), A le deja un mensaje a RabbitMQ. B lo recoge cuando esté disponible. Los dos servicios no necesitan conocerse ni estar activos al mismo tiempo.

📬

Message Broker

Intermediario que recibe, enruta y entrega mensajes entre aplicaciones desacopladas.

⚡

Alto Rendimiento

Decenas de miles de mensajes/seg. Erlang le da concurrencia masiva y tolerancia a fallos.

🔀

Enrutamiento flexible

Exchanges con múltiples estrategias: direct, fanout, topic, headers.

✅

ACKs garantizados

Confirmaciones de entrega en ambos sentidos: publisher confirms y consumer acks.

¿Por qué mensajería asíncrona?

En arquitecturas de microservicios, la comunicación síncrona HTTP crea acoplamiento temporal: si el servicio B está caído, A falla. La mensajería asíncrona rompe ese acoplamiento.

Comunicación Síncrona (HTTP)
  • A llama a B y espera respuesta
  • Si B falla, A también falla
  • Escala acoplado
  • Latencia de red directa
  • Ideal: operaciones de lectura, APIs públicas
Mensajería Asíncrona (RabbitMQ) ✅
  • A publica mensaje y continúa
  • Si B falla, el mensaje espera en la queue
  • Productores y consumidores escalan independientemente
  • Buffer natural ante picos de tráfico
  • Ideal: operaciones de escritura, workflows, tasks

RabbitMQ vs Kafka — ¿Cuándo usar cuál?

Criterio RabbitMQ Kafka
Modelo Message queue (broker inteligente) Distributed log (broker tonto)
Retención Mensaje se elimina al hacer ACK Mensaje persiste N días
Enrutamiento Muy flexible (exchanges, bindings) Simple (topic + partition key)
Orden FIFO por queue, no global Orden garantizado por partición
Throughput ~50K msg/s por nodo Millones msg/s
Latencia Sub-milisegundo Milisegundos
Ideal para Tasks, workflows, RPC, routing complejo Streaming, analytics, event sourcing

Casos de uso reales

  • Procesamiento de pedidos: el servicio web publica el pedido, servicios de inventario/pago/notificación los consumen independientemente.
  • Envío de emails/SMS: la app encola mensajes, workers los procesan sin bloquear la respuesta HTTP.
  • Microservicios RPC: llamadas asíncronas con reply queues para respuestas.
  • Rate limiting: las queues actúan como buffer ante picos de tráfico.
  • Workflows complejos: mensajes enrutados a diferentes servicios según su contenido.
🐇
Regla simple: Si necesitas enrutamiento inteligente, confirmación de entrega garantizada, o trabajas con tareas discretas y workflows — elige RabbitMQ. Si necesitas replay de eventos, analytics o millones de mensajes/segundo — elige Kafka.

Arquitectura AMQP

El protocolo base: conexiones, canales y el modelo de mensajería

RabbitMQ implementa el protocolo AMQP 0-9-1. Antes de escribir código, es importante entender las piezas que componen este modelo: cada una tiene un rol claro, y juntas forman el camino que recorre un mensaje desde que se publica hasta que se procesa.

Componentes del modelo AMQP

📤

Producer

Aplicación que publica mensajes al broker. Solo habla con exchanges, nunca con queues directamente.

🔀

Exchange

Recibe mensajes del producer y los enruta a queues según reglas (bindings). El "cerebro" del enrutamiento.

📦

Queue

Buffer donde los mensajes esperan ser consumidos. FIFO. Puede ser durable (persiste reinicios) o transient.

🔗

Binding

Regla que conecta un Exchange con una Queue, con una routing key opcional.

📥

Consumer

Aplicación que se suscribe a una queue y recibe mensajes. Confirma con ACK o rechaza con NACK.

🏠

Virtual Host

Namespace lógico de aislamiento. Cada vhost tiene sus propios exchanges, queues y usuarios.

Flujo completo de un mensaje

Ciclo de vida de un mensaje AMQP
[Producer] │ 1. Abre Connection TCP al broker │ 2. Abre Channel (multiplexado sobre la connection) │ 3. Publica mensaje a un Exchange + routing_key ▼ [Exchange] │ 4. Evalúa las reglas de binding │ 5. Enruta el mensaje a la(s) queue(s) correspondiente(s) ▼ [Queue] │ 6. Almacena el mensaje (en RAM o disco si es persistent) │ 7. Aguarda hasta que un consumer esté disponible ▼ [Consumer] │ 8. Recibe el mensaje vía push (subscribe) o pull (basic.get) │ 9. Procesa el mensaje │ 10. Envía ACK → broker elimina el mensaje └ (o NACK → broker reencola o descarta)

Connection vs Channel

Esta distinción es crítica para el rendimiento. Abrir una conexión TCP por cada operación es muy costoso. AMQP soluciona esto con channels.

Connection (TCP)
  • Conexión TCP real al broker
  • Costosa de abrir (handshake)
  • 1 por aplicación (o pocas)
  • Mantenida con heartbeats
Channel (virtual) ✅
  • Canal virtual sobre la Connection
  • Ligero, barato de crear
  • 1 por goroutine/thread
  • Permite operaciones paralelas
⚠️
Anti-pattern: Nunca compartas un channel entre goroutines — no es thread-safe. Usa 1 channel por goroutine. La Connection sí puede compartirse.

Propiedades de un mensaje

Propiedad Tipo Descripción
content_type string MIME type del cuerpo: application/json, text/plain
delivery_mode 1 o 2 1 = transient (solo RAM), 2 = persistent (disco)
priority 0-255 Prioridad del mensaje (requiere queue con max-priority)

Exchanges y Tipos

El corazón del enrutamiento: Direct, Fanout, Topic y Headers

El Exchange es la pieza que decide adónde va cada mensaje. El producer nunca escribe directamente en una queue — siempre le entrega el mensaje al exchange, que lo enruta usando sus reglas de binding. Hay cuatro tipos de exchange, cada uno con una lógica de enrutamiento distinta. Elegir el tipo correcto define la arquitectura de tu sistema de mensajería.

Direct Exchange

Enruta mensajes a queues cuya binding key coincide exactamente con la routing key del mensaje.

Direct Exchange — coincidencia exacta
Producer ──publish(routing_key="error")──▶ [Direct Exchange: logs] │ binding_key="error" ──────────▶ [Queue: errores] → Consumer-A binding_key="info" ──────────▶ [Queue: info] ← no recibe este mensaje binding_key="warn" ──────────▶ [Queue: warnings]← no recibe este mensaje → Solo la queue "errores" recibe el mensaje porque su binding_key = "error"

Fanout Exchange

Ignora la routing key. Difunde el mensaje a TODAS las queues vinculadas. Broadcast puro.

Fanout Exchange — broadcast
Producer ──publish(routing_key ignorada)──▶ [Fanout Exchange: pedidos] │ ┌────────────────┼────────────────┐ ▼ ▼ ▼ [Queue: inventario] [Queue: pagos] [Queue: notificaciones] Consumer-A Consumer-B Consumer-C → Los 3 consumers reciben el mismo mensaje simultáneamente

Topic Exchange

Enrutamiento con wildcards en la routing key. Usa puntos como separadores. El más flexible.

Wildcard Significado Ejemplo
* Exactamente una palabra orden.* coincide con orden.creada pero NO con orden.pago.procesado
# Cero o más palabras orden.# coincide con orden.creada, orden.pago.procesado, orden
Topic Exchange — routing con wildcards
Routing keys de mensajes: "orden.creada", "orden.pagada", "orden.enviada", "usuario.registrado" [Topic Exchange: eventos] binding "orden.*" → [Queue: auditoria-ordenes] recibe: orden.creada, orden.pagada, orden.enviada binding "orden.creada" → [Queue: inventario] recibe: orden.creada (solo) binding "#" → [Queue: logs-global] recibe: TODOS los mensajes binding "usuario.*" → [Queue: bienvenida] recibe: usuario.registrado

Headers Exchange

Ignora la routing key. Enruta basándose en los headers del mensaje. Usa x-match: all (todos los headers deben coincidir) o x-match: any (cualquiera).

bash headers exchange ejemplo
# Binding con headers
# Queue "pagos-tarjeta" se activa si el mensaje tiene:
# header "metodo_pago" = "tarjeta" AND "region" = "colombia"

# x-match: all → TODOS los headers deben coincidir
# x-match: any → AL MENOS UNO debe coincidir

# Mensaje con headers: {metodo_pago: "tarjeta", region: "colombia"}
# → Llega a "pagos-tarjeta" ✅

# Mensaje con headers: {metodo_pago: "efectivo", region: "colombia"}
# → NO llega a "pagos-tarjeta" ❌ (metodo_pago no coincide)

Default Exchange

Exchange especial sin nombre (""). Todo queue creado tiene automáticamente un binding con el Default Exchange, donde la routing key es el nombre de la queue.

💡
Publicar a exchange="" con routing_key="mi-queue" equivale a enviar directamente a la queue "mi-queue". Es el patrón más simple para empezar.

Resumen de elección

Tipo Cuándo usar Ejemplo de uso
Direct Enrutamiento a queues específicas por categoría exacta Niveles de log, tipos de tarea
Fanout Notificar a múltiples servicios el mismo evento Evento "orden creada" → inventario, pagos, notificaciones
Topic Enrutamiento flexible con jerarquías y wildcards Sistema de eventos multi-dominio
Headers Enrutamiento basado en metadata, sin routing key Enrutar por idioma, región, tipo de cliente

Queues y Bindings

Propiedades, tipos especiales y cómo conectar exchanges con queues

La queue es el almacén de mensajes: guarda los mensajes hasta que un consumidor los recoja. Los bindings son las reglas que conectan un exchange con una queue. Sin bindings, los mensajes que llegan al exchange no tienen a dónde ir y se pierden. Configurar bien estas dos piezas es esencial para que el sistema funcione como esperas.

Propiedades de una Queue

Propiedad Valores Descripción
Durable true/false Si true, la queue sobrevive reinicios del broker. Los mensajes dentro también deben ser persistent.
Exclusive true/false Solo puede usarla la conexión que la creó. Se elimina cuando esa conexión cierra. Ideal para queues temporales de respuesta.
Auto-delete true/false Se elimina automáticamente cuando el último consumer se desconecta.
Arguments map Argumentos adicionales: TTL, DLX, max-length, priority, etc.

Argumentos (x-arguments) importantes

bash declarar queue con argumentos
# Queue con TTL de mensajes (30 segundos)
rabbitmqadmin declare queue \
  name=pedidos-temporales \
  durable=true \
  arguments='{"x-message-ttl": 30000}'

# Queue con máximo de mensajes (si se llena, descarta los más viejos)
rabbitmqadmin declare queue \
  name=metricas \
  durable=true \
  arguments='{"x-max-length": 10000, "x-overflow": "drop-head"}'

# Queue con Dead Letter Exchange (para mensajes fallidos)
rabbitmqadmin declare queue \
  name=pedidos \
  durable=true \
  arguments='{"x-dead-letter-exchange": "dlx", "x-dead-letter-routing-key": "pedidos.failed"}'

# Queue con prioridad (0-255, necesita x-max-priority)
rabbitmqadmin declare queue \
  name=tareas-prioritarias \
  durable=true \
  arguments='{"x-max-priority": 10}'

# Queue con expiración (se elimina si no tiene consumers en N ms)
rabbitmqadmin declare queue \
  name=queue-temporal \
  arguments='{"x-expires": 60000}'

Tipos de Queue (RabbitMQ 3.8+)

📋

Classic Queue

El tipo original. Rápida pero sin replicación automática entre nodos del cluster.

🗳️

Quorum Queue

Replicada entre N nodos usando Raft. La recomendada para producción. Durabilidad garantizada.

🌊

Stream Queue

Append-only log (similar a Kafka). Permite replay desde cualquier offset. Disponible en 3.9+.

🏆
Recomendación de producción: Usa Quorum Queues para datos críticos (tolerancia a fallos automática) y Classic Queues para datos no críticos donde el rendimiento importa más.

Bindings

bash gestión de bindings

  source=mi-exchange \
  destination=mi-queue \
  routing_key=mi-routing-key

# Binding con Topic Exchange y wildcard
rabbitmqadmin declare binding \
  source=eventos \
  destination=auditoria \

# Binding con Fanout (routing_key se ignora)
rabbitmqadmin declare binding \
  source=broadcast \
  destination=servicio-a \

# Listar bindings
rabbitmqadmin list bindings

# Eliminar binding
rabbitmqadmin delete binding \
  source=mi-exchange \
  destination=mi-queue \

Exchange-to-Exchange Bindings

RabbitMQ permite vincular exchanges entre sí, creando topologías de enrutamiento complejas:

Exchange → Exchange → Queue
Producer ──▶ [Topic Exchange: eventos.globales] │ binding: "orden.*" ▼ [Fanout Exchange: eventos.ordenes] │ │ │ ▼ ▼ ▼ [Q: inventario] [Q: pagos] [Q: notificaciones] → Filtrar primero por Topic, luego broadcast via Fanout

Publicar mensajes: confirmaciones, propiedades y mejores prácticas

📌

Si aún no tienes el proyecto configurado, no te preocupes — en el Módulo 11 (RabbitMQ con Go) se hace el setup completo. Por ahora, enfócate en entender los conceptos de publisher confirms, propiedades del mensaje y el patrón de publicación. El código completo lo ejecutarás en el módulo 11.

El producer es cualquier parte de tu aplicación que necesita enviar un mensaje. Para publicar de forma segura en producción hay que configurar dos cosas clave: las propiedades del mensaje (qué datos lleva, cómo de importante es) y los publisher confirms (para saber con certeza que el broker lo recibió).

Publisher Confirms

Por defecto, basic.publish es fire-and-forget — no sabes si el broker recibió el mensaje. Publisher Confirms activa confirmaciones asíncronas del broker.

Flujo de Publisher Confirms
Producer Broker │ 1. channel.Confirm() ──────────────────▶ │ Activa modo confirm │ │ │ 2. Publish(msg #1) ──────────────────▶ │ Recibe y persiste │ 3. Publish(msg #2) ──────────────────▶ │ Recibe y persiste │ │ │ ◀──── basic.ack(deliveryTag=1) ─── │ Confirma msg #1 │ ◀──── basic.ack(deliveryTag=2) ─── │ Confirma msg #2 │ │ │ (si falla) ◀─── basic.nack(tag=N) ──────── │ Nack = reintentar

Propiedades del mensaje — profundidad

Cada mensaje AMQP lleva dos partes: el cuerpo (body, tus datos) y las propiedades, que son metadatos estándar que van separados. Las propiedades describen el contenido (ContentType), controlan la durabilidad (DeliveryMode), habilitan el patrón RPC (CorrelationId, ReplyTo), añaden trazabilidad distribuida (MessageId, AppId, headers personalizados) y permiten definir expiración (Expiration). Usarlas correctamente marca la diferencia entre un sistema básico y uno listo para producción.

go publicar con propiedades completas
// Publicar mensaje con todas las propiedades relevantes
err = ch.PublishWithContext(ctx,
    "mi-exchange",          // exchange
    "orden.creada",         // routing key
    true,                   // mandatory: true = error si no hay queue destino
    false,                  // immediate: deprecated en RabbitMQ 3.x, usar false
    amqp.Publishing{
        // Identificación
        MessageId:     uuid.New().String(),
        CorrelationId: req.CorrelationID,   // para RPC pattern
        ReplyTo:       replyQueue,          // para RPC pattern

        // Contenido
        ContentType:   "application/json",
        ContentEncoding: "utf-8",
        Body:          jsonBytes,

        // Durabilidad: 1=transient, 2=persistent
        DeliveryMode: amqp.Persistent,

        // Expiración (en ms, como string)
        Expiration: "60000", // 1 minuto

        // Prioridad (0-9 si la queue tiene x-max-priority)
        Priority: 5,

        // Headers personalizados
        Headers: amqp.Table{
            "source-service": "pedidos-service",
            "version":        "1.0",
            "trace-id":       traceID,
        },

        // Timestamp
        Timestamp: time.Now(),
        Type:      "orden.creada.v1",
        AppId:     "pedidos-service",
    },
)

Patrón: Producer robusto con confirmaciones

Las Publisher Confirms garantizan que el broker recibió y persistió el mensaje — pero su API es asíncrona. El patrón idiomático en Go es: activar channel.Confirm(false), suscribirse al canal NotifyPublish, publicar el mensaje y luego bloquear con select esperando el ACK del broker. Si llega un NACK o el timeout expira, el producer puede decidir reintentar o registrar el error para inspección manual. Este patrón también debe suscribirse a NotifyReturn para capturar mensajes devueltos porque no encontraron queue de destino.

go producer con publisher confirms
type Producer struct {
    conn    *amqp.Connection
    channel *amqp.Channel
    confirms chan amqp.Confirmation
}

func NewProducer(url string) (*Producer, error) {
    conn, err := amqp.Dial(url)
    if err != nil { return nil, fmt.Errorf("conectar RabbitMQ: %w", err) }

    ch, err := conn.Channel()
    if err != nil { return nil, err }

    // Activar publisher confirms
    if err = ch.Confirm(false); err != nil {
        return nil, fmt.Errorf("activar confirms: %w", err)
    }

    // Canal para recibir confirmaciones (buffer 100)
    confirms := ch.NotifyPublish(make(chan amqp.Confirmation, 100))

    return &Producer{conn: conn, channel: ch, confirms: confirms}, nil
}

func (p *Producer) Publish(ctx context.Context, exchange, key string, body []byte) error {
    err := p.channel.PublishWithContext(ctx, exchange, key, true, false,
        amqp.Publishing{
            ContentType:  "application/json",
            Body:         body,
            DeliveryMode: amqp.Persistent,
            MessageId:    uuid.New().String(),
            Timestamp:    time.Now(),
        },
    )
    if err != nil { return fmt.Errorf("publicar: %w", err) }

    // Esperar confirmación del broker con timeout
    select {
    case confirm := <-p.confirms:
        if !confirm.Ack {
            return fmt.Errorf("broker rechazó el mensaje (nack)")
        }
        return nil
    case <-time.After(5 * time.Second):
        return fmt.Errorf("timeout esperando confirmación del broker")
    case <-ctx.Done():
        return ctx.Err()
    }
}

func (p *Producer) Close() {
    p.channel.Close()
    p.conn.Close()
}
⚠️
Mandatory flag: Si mandatory=true y el mensaje no llega a ninguna queue (no hay binding), el broker devuelve el mensaje al producer via NotifyReturn. Siempre suscríbete a este canal para detectar mensajes perdidos.

Consumers

Suscripción, prefetch, push vs pull y patrones de consumo

El consumer es quien recibe los mensajes de una queue y los procesa. Configurarlo bien es fundamental: un consumer mal configurado puede sobrecargar tu aplicación, procesar mensajes en orden incorrecto o, lo más crítico, perder mensajes si el procesamiento falla antes de confirmar la recepción.

Push vs Pull

Push — basic.consume ✅ Recomendado
  • El broker empuja mensajes al consumer
  • Alta eficiencia y baja latencia
  • Consumer permanece suscrito
  • El broker controla el flujo con prefetch
  • Ideal para la mayoría de casos
Pull — basic.get ⚠️ Usar con cuidado
  • Consumer pide mensajes explícitamente
  • Polling = ineficiente si la queue está vacía
  • Útil para procesar 1 mensaje a la vez
  • No recomendado para alto throughput

Prefetch Count — Control de flujo

El prefetch_count limita cuántos mensajes sin ACK puede tener un consumer simultáneamente. Es la configuración más importante para el rendimiento y equidad.

Sin prefetch (prefetch=0, sin límite) — ¡PROBLEMA!
Queue: [msg1, msg2, ..., msg1000] Consumer-A (rápido) ← recibe 900 mensajes de golpe Consumer-B (lento) ← recibe 100 mensajes → Consumer-B sobrecargado, Consumer-A ocioso. Desequilibrado.
Con prefetch=1 — distribución equitativa
Queue: [msg1, msg2, msg3, msg4, msg5...] Consumer-A procesa msg1 → ACK → recibe msg3 → ACK → recibe msg5... Consumer-B procesa msg2 → ACK → recibe msg4 → ACK → recibe msg6... → Cada consumer recibe el siguiente mensaje solo cuando termina el anterior
📐
Guía de prefetch: prefetch=1 para tareas pesadas (máxima equidad). prefetch=10-50 para tareas ligeras (mayor throughput). Nunca prefetch=0 en producción.

Consumer Tags y Cancelación

El consumer tag es un identificador único dentro del canal que permite cancelar un consumer específico con basic.cancel sin cerrar el canal ni afectar a otros consumers activos. Al llamar a ch.Consume recibes un canal Go <-chan amqp.Delivery que entrega los mensajes de forma concurrente. El loop con select sobre ese canal y ctx.Done() es el patrón estándar para consumo cancelable con context: cuando el context se cancela (por shutdown o timeout), el consumer termina limpiamente.

go consumer básico con prefetch
func (c *Consumer) Start(ctx context.Context) error {
    // Configurar prefetch ANTES de consumir
    if err := c.ch.Qos(
        10,    // prefetch_count: máx mensajes sin ACK
        0,     // prefetch_size: 0 = sin límite por tamaño
        false, // global: false = por consumer, true = por canal
    ); err != nil {
        return fmt.Errorf("configurar QoS: %w", err)
    }

    // Suscribirse a la queue (modo push)
    msgs, err := c.ch.Consume(
        "mi-queue",        // queue
        "worker-1",        // consumer tag (identificador único)
        false,             // auto-ack: FALSE = manual (recomendado)
        false,             // exclusive: solo este consumer en la queue
        false,             // no-local: ignorar mensajes del mismo conn
        false,             // no-wait: false = esperar confirmación
        nil,               // args adicionales
    )
    if err != nil { return fmt.Errorf("consumir: %w", err) }

    log.Println("Consumer iniciado, esperando mensajes...")

    for {
        select {
        case msg, ok := <-msgs:
            if !ok {
                return fmt.Errorf("canal cerrado inesperadamente")
            }
            if err := c.handleMessage(ctx, msg); err != nil {
                log.Printf("error procesando msg: %v", err)
                // NACK sin requeue → va a DLQ
                msg.Nack(false, false)
                continue
            }
            // ACK solo si el procesamiento fue exitoso
            msg.Ack(false) // false = solo este mensaje (no multiple)
        case <-ctx.Done():
            return nil
        }
    }
}

Acknowledgements y Durabilidad

Garantías de entrega, ACK, NACK, Reject y persistencia de mensajes

El sistema de acknowledgements (confirmaciones) de RabbitMQ es lo que garantiza que ningún mensaje se pierda. El principio es sencillo: el broker considera que un mensaje sigue pendiente hasta que el consumer le confirma con un ACK que lo procesó correctamente. Si el consumer falla antes de confirmar, el broker lo reenvía automáticamente.

Tipos de ACK

Operación Efecto Cuándo usar
Ack(multiple=false) Confirma exactamente este mensaje. Broker lo elimina. Procesamiento exitoso normal.
Ack(multiple=true) Confirma este y todos los mensajes anteriores en el canal. Procesamiento en batch para mayor throughput.
Nack(multiple, requeue=true) Rechaza el mensaje y lo reencola al frente. Error temporal (DB no disponible, timeout).
Nack(multiple, requeue=false) Rechaza el mensaje y lo descarta (o va a DLX). Error permanente (JSON inválido, regla de negocio).
Reject(requeue=true) Igual que Nack pero para un solo mensaje. Mismo que Nack individual.
Reject(requeue=false) Descarta un mensaje individual. Mensaje imposible de procesar.
🚨
Infinite loop de NACK: Si haces Nack(requeue=true) en un error permanente, el mensaje vuelve a la queue y el mismo consumer lo vuelve a fallar infinitamente. Usa requeue=false para errores permanentes y configura un DLX para capturarlos.

Estados de un mensaje

Ciclo de vida del mensaje en el consumer
Queue: [msg] │ ▼ (broker entrega al consumer) Estado: unacked ← el mensaje existe en broker Y en consumer │ ├── Ack() → eliminado del broker ✅ éxito │ ├── Nack(requeue=true) → vuelve a la queue ↩️ reintentar │ ├── Nack(requeue=false) → descartado / va a DLX 🗑️ error permanente │ └── consumer se desconecta → vuelve a la queue automáticamente ↩️ redelivered=true

Durabilidad completa — checklist

  1. Exchange declarado con durable=true
  2. Queue declarada con durable=true
  3. Mensaje publicado con delivery_mode=2 (persistent)
  4. Publisher Confirms activados (channel.Confirm(false))
  5. Consumer con auto-ack=false (ACK manual)

Manejo de redeliveries

Cuando un consumer se desconecta sin hacer ACK o hace Nack(requeue=true), el broker reenvía el mensaje con el flag Redelivered=true. Esto puede provocar procesamiento duplicado si no se maneja. Existen dos estrategias: idempotencia (diseñar la operación para que ejecutarla varias veces produzca el mismo resultado) y deduplicación activa (guardar los MessageId ya procesados y hacer ACK sin reprocesar si ya fue visto). El campo x-retry-count en los headers es un mecanismo manual para limitar reintentos antes de enviar definitivamente a la DLQ.

func handleMessage(msg amqp.Delivery) error {
    // Detectar si es un reenvío (ya fue entregado antes)
    if msg.Redelivered {
        log.Printf("⚠️  Mensaje reenviado: %s", msg.MessageId)
        // Estrategia 1: procesar de todas formas (idempotencia)
        // Estrategia 2: limitar reintentos con headers
    }

    // Contar reintentos con headers personalizados
    retryCount := 0

        // Republicar con contador incrementado
        return republishWithRetry(msg, retryCount+1)
    }

}

Patrones de Mensajería

Work Queue (Task Queue)

Distribuye tareas pesadas entre múltiples workers. El broker balancea la carga automáticamente.

Producer ──▶ [Default Exchange] ──▶ [Queue: tareas] │ ┌───────────────────────┤ ▼ ▼ [Worker-1] [Worker-2] (procesa task-1) (procesa task-2) Con prefetch=1: distribución equitativa automática

Publish/Subscribe (Pub/Sub)

Un evento notifica simultáneamente a múltiples servicios independientes.

Producer ──▶ [Fanout Exchange: orden.eventos] │ ┌───────────────────┼────────────────────┐ ▼ ▼ ▼ [Q: inventario] [Q: facturacion] [Q: notificacion] (reduce stock) (genera factura) (envía email) Cada servicio tiene su propia queue — procesamiento independiente

Routing (Direct Exchange)

Producer ──error──▶ [Direct Exchange: logs] ──error──▶ [Q: alertas-criticas] ──info───▶ ──info───▶ [Q: logs-generales] ──warning▶ ──warning▶ [Q: logs-generales] alertas-criticas: solo bindings "error" logs-generales: bindings "info", "warning"

RPC (Remote Procedure Call)

Llamadas síncronas sobre mensajería asíncrona. Útil para obtener respuestas de servicios.

Client (Producer+Consumer) Server (Consumer+Producer) 1. Crea queue temporal: reply_to="amq.gen-XYZ" 2. Competing Consumers (escalado)

En este patrón todos los workers consumen de la misma queue — el broker distribuye los mensajes entre ellos automáticamente. Es diferente al Pub/Sub donde cada suscriptor tiene su propia queue. El escalado es horizontal: agregas más workers sin modificar el producer ni la topología de exchanges y queues. Con prefetch=1, la distribución es equitativa aunque los workers procesen a velocidades distintas. Cada worker necesita su propio Channel, pero todos comparten la misma Connection.

go múltiples consumers compitiendo
// Lanzar N workers compitiendo por la misma queue
func StartWorkerPool(ctx context.Context, conn *amqp.Connection, n int) {
    var wg sync.WaitGroup
    for i := 0; i < n; i++ {
        wg.Add(1)
        go func(workerID int) {
            defer wg.Done()
            // Cada worker necesita su propio channel
            ch, _ := conn.Channel()
            defer ch.Close()
            ch.Qos(1, 0, false) // prefetch=1 por worker

            msgs, _ := ch.Consume("tareas", fmt.Sprintf("worker-%d", workerID),
                false, false, false, false, nil)

            for {
                select {
                case msg, ok := <-msgs:
                    if !ok { return }
                    log.Printf("[Worker-%d] procesando %s", workerID, msg.MessageId)
                    if err := procesarTarea(msg.Body); err != nil {
                        msg.Nack(false, false) // DLQ
                    } else {
                        msg.Ack(false)
                    }
                case <-ctx.Done():
                    return
                }
            }
        }(i)
    }
    wg.Wait()
}

Dead Letter Queues

DLX, TTL, retry con backoff y manejo de mensajes fallidos

En cualquier sistema de mensajería hay mensajes que fallan: el servicio externo no responde, el JSON llega malformado, o hay un bug en la lógica de negocio. Sin una estrategia para manejar estos fallos, los mensajes simplemente desaparecen. La Dead Letter Queue es la red de seguridad: captura esos mensajes para que puedas inspeccionarlos y reprocesarlos sin perderlos.

¿Cuándo un mensaje se convierte en "dead letter"?

  • El consumer hace Nack(requeue=false) o Reject(requeue=false)
  • El mensaje expira por TTL (x-message-ttl)
  • La queue alcanza su límite máximo (x-max-length) y se descarta el más viejo

Configuración DLX

Configurar un Dead Letter Exchange requiere declarar cuatro componentes en orden: el exchange DLX (normalmente direct), la queue que recibirá los mensajes fallidos (la DLQ), el binding entre ambos, y finalmente tu queue principal con el argumento x-dead-letter-exchange apuntando al DLX. Cuando un mensaje muere en la queue principal (por NACK, TTL expirado o max-length superado), RabbitMQ lo re-publica automáticamente al DLX usando la routing key definida en x-dead-letter-routing-key, y desde ahí llega a la DLQ para inspección o reprocesamiento manual.

go configurar DLX completo
func setupDLX(ch *amqp.Channel) error {
    // 1. Declarar el Dead Letter Exchange
    err := ch.ExchangeDeclare("dlx", "direct", true, false, false, false, nil)
    if err != nil { return err }

    // 2. Declarar la Dead Letter Queue (donde van los mensajes fallidos)
    _, err = ch.QueueDeclare("pedidos.dlq", true, false, false, false,
        amqp.Table{
            "x-message-ttl": int64(86400000), // mensajes en DLQ expiran en 24h
        },
    )
    if err != nil { return err }

    // 3. Binding: DLX → DLQ
    err = ch.QueueBind("pedidos.dlq", "pedidos.failed", "dlx", false, nil)
    if err != nil { return err }

    // 4. Declarar la queue PRINCIPAL con referencia al DLX
    _, err = ch.QueueDeclare("pedidos", true, false, false, false,
        amqp.Table{
            "x-dead-letter-exchange":    "dlx",
            "x-dead-letter-routing-key": "pedidos.failed",
            // opcional: TTL de mensajes en cola principal
            "x-message-ttl": int64(300000), // 5 minutos
        },
    )
    return err
}

go retry con conteo de intentos
const maxRetries = 3

func processWithRetry(msg amqp.Delivery, producer *Producer) error {
    // Extraer conteo de muertes del header x-death
    retryCount := getDeathCount(msg.Headers)

    if retryCount >= maxRetries {
        log.Printf("❌ Agotados reintentos para %s", msg.MessageId)
        return msg.Nack(false, false) // Va a DLQ final
    }

    if err := procesarNegocio(msg.Body); err != nil {
        delay := time.Duration(30*(retryCount+1)) * time.Second
        log.Printf("⚠️  Fallo #%d, reintento en %v", retryCount+1, delay)
        return msg.Nack(false, false) // Va a queue de retry con TTL
    }
    return msg.Ack(false)
}

func getDeathCount(headers amqp.Table) int {
    xDeath, ok := headers["x-death"]
    if !ok { return 0 }
    deaths, ok := xDeath.([]interface{})
    if !ok || len(deaths) == 0 { return 0 }
    death := deaths[0].(amqp.Table)
    count, _ := death["count"].(int64)
    return int(count)
}

Cliente base con reconexión automática

En producción las conexiones TCP pueden caerse por reinicios del broker, cortes de red o timeouts. La solución correcta es un Client que gestione la conexión de forma inteligente: reintenta con backoff exponencial al conectar (1s → 2s → 4s → 8s...) y monitorea la conexión en segundo plano con una goroutine para reconectar si se cae. La Connection AMQP es thread-safe y puede compartirse entre goroutines; los Channel no.

go internal/rmq/client.go
package rmq

import (
    "context"
    "fmt"
    "log/slog"
    "time"

    amqp "github.com/rabbitmq/amqp091-go"
)

type Client struct {
    url     string
    conn    *amqp.Connection
    logger  *slog.Logger
}

func NewClient(url string) *Client {
    return &Client{url: url, logger: slog.Default()}
}

// Connect con reintentos exponenciales
func (c *Client) Connect(ctx context.Context) error {
    delays := []time.Duration{1, 2, 4, 8, 16, 32}
    for i, d := range delays {
        conn, err := amqp.Dial(c.url)
        if err == nil {
            c.conn = conn
            c.logger.Info("conectado a RabbitMQ")
            go c.watchConnection(ctx) // monitorear desconexiones
            return nil
        }
        c.logger.Warn("fallo al conectar, reintentando", "attempt", i+1, "delay", d)
        select {
        case <-time.After(d * time.Second):
        case <-ctx.Done():
            return ctx.Err()
        }
    }
    return fmt.Errorf("no se pudo conectar a RabbitMQ después de %d intentos", len(delays))
}

// watchConnection detecta desconexiones y reconecta
func (c *Client) watchConnection(ctx context.Context) {
    notifyClose := c.conn.NotifyClose(make(chan *amqp.Error, 1))
    select {
    case err := <-notifyClose:
        if err != nil {
            c.logger.Error("conexión RabbitMQ cerrada inesperadamente", "err", err)
            c.Connect(ctx) // reconectar
        }
    case <-ctx.Done():
    }
}

func (c *Client) NewChannel() (*amqp.Channel, error) {
    if c.conn == nil || c.conn.IsClosed() {
        return nil, fmt.Errorf("conexión no disponible")
    }
    return c.conn.Channel()
}

func (c *Client) Close() {
    if c.conn != nil && !c.conn.IsClosed() {
        c.conn.Close()
    }
}

    ch, err := client.NewChannel()
    if err != nil { return nil, err }

    // Activar publisher confirms
    if err = ch.Confirm(false); err != nil {
        return nil, fmt.Errorf("activar confirms: %w", err)
    }

    confirms := ch.NotifyPublish(make(chan amqp.Confirmation, 100))

    // Capturar mensajes no enrutados
    returns := ch.NotifyReturn(make(chan amqp.Return, 10))
    go func() {
        for r := range returns {
            slog.Warn("mensaje no enrutado devuelto",
                "exchange", r.Exchange,
                "routing_key", r.RoutingKey,
                "reply_code", r.ReplyCode,
                "reply_text", r.ReplyText,
            )
        }
    }()

    return &Producer{client: client, ch: ch, confirms: confirms}, nil
}

// Publish serializa a JSON y publica con confirmación
func (p *Producer) Publish(ctx context.Context, exchange, routingKey string, payload any) error {
    body, err := json.Marshal(payload)
    if err != nil { return fmt.Errorf("serializar: %w", err) }

    err = p.ch.PublishWithContext(ctx,
        exchange, routingKey,
        true,  // mandatory
        false, // immediate
        amqp.Publishing{
            MessageId:    uuid.New().String(),
            ContentType:  "application/json",
            DeliveryMode: amqp.Persistent,
            Timestamp:    time.Now(),
            Body:         body,
        },
    )
    if err != nil { return fmt.Errorf("publicar: %w", err) }

    // Esperar ack del broker
    select {
    case confirm := <-p.confirms:
        if !confirm.Ack {
            return fmt.Errorf("broker no confirmó el mensaje")
        }
        return nil
    case <-time.After(10 * time.Second):
        return fmt.Errorf("timeout en publisher confirm")
    case <-ctx.Done():
        return ctx.Err()
    }
}

func (p *Producer) Close() { p.ch.Close() }

Consumer de producción con goroutines

El Consumer sigue el patrón de handler inyectable: acepta una función Handler(ctx, msg) error con la lógica de negocio, y la infraestructura (prefetch, ACK/NACK, recuperación de panics) queda encapsulada en el Consumer. La regla es simple: si el handler retorna nil → ACK (procesado con éxito); si retorna error → NACK sin requeue (va a la DLQ). El recover() envuelve cada mensaje para que un panic no derribe el worker completo. StartPool lanza N goroutines independientes, cada una con su propio channel.

go internal/rmq/consumer.go
package rmq

import (
    "context"
    "fmt"
    "log/slog"

    amqp "github.com/rabbitmq/amqp091-go"
)

type Handler func(ctx context.Context, msg amqp.Delivery) error

type Consumer struct {
    client  *Client
    queue   string
    tag     string
    handler Handler
}

func NewConsumer(client *Client, queue, tag string, handler Handler) *Consumer {
    return &Consumer{client: client, queue: queue, tag: tag, handler: handler}
}

func (c *Consumer) Start(ctx context.Context) error {
    ch, err := c.client.NewChannel()
    if err != nil { return err }
    defer ch.Close()

    // Prefetch: 1 mensaje a la vez por worker
    if err = ch.Qos(1, 0, false); err != nil {
        return fmt.Errorf("configurar QoS: %w", err)
    }

    msgs, err := ch.Consume(c.queue, c.tag, false, false, false, false, nil)
    if err != nil { return fmt.Errorf("consumir: %w", err) }

    slog.Info("consumer iniciado", "queue", c.queue, "tag", c.tag)

    for {
        select {
        case msg, ok := <-msgs:
            if !ok {
                return fmt.Errorf("canal cerrado")
            }
            c.process(ctx, msg)
        case <-ctx.Done():
            slog.Info("consumer detenido", "queue", c.queue)
            return nil
        }
    }
}

func (c *Consumer) process(ctx context.Context, msg amqp.Delivery) {
    defer func() {
        if r := recover(); r != nil {
            slog.Error("panic en handler", "panic", r)
            msg.Nack(false, false) // DLQ
        }
    }()

    slog.Debug("procesando mensaje",
        "message_id", msg.MessageId,
        "queue", c.queue,
        "redelivered", msg.Redelivered,
    )

    if err := c.handler(ctx, msg); err != nil {
        slog.Error("error en handler", "err", err, "message_id", msg.MessageId)
        msg.Nack(false, false)
        return
    }
    msg.Ack(false)
}

// StartPool lanza N consumers compitiendo
func (c *Consumer) StartPool(ctx context.Context, n int) {
    for i := 0; i < n; i++ {
        go func(id int) {
            worker := &Consumer{
                client:  c.client,
                queue:   c.queue,
                tag:     fmt.Sprintf("%s-%d", c.tag, id),
                handler: c.handler,
            }
            if err := worker.Start(ctx); err != nil {
                slog.Error("worker terminó con error", "id", id, "err", err)
            }
        }(i)
    }
}

main.go — integración completa

El main.go ensambla todos los componentes: crea el cliente, el producer y arranca el pool de consumers. Usa signal.NotifyContext para capturar SIGINT/SIGTERM y cancelar el contexto cuando el proceso se apaga — esa cancelación se propaga automáticamente a todos los consumers y goroutines activas, garantizando un cierre limpio sin perder mensajes en vuelo.

go cmd/main.go
package main

import (
    "context"
    "encoding/json"
    "log/slog"
    "os/signal"
    "syscall"

    amqp "github.com/rabbitmq/amqp091-go"
    "github.com/elkin/rabbitmq-service/internal/rmq"
)

type OrdenCreada struct {
    OrdenID   string  `json:"orden_id"`
    UsuarioID string  `json:"usuario_id"`
    Total     float64 `json:"total"`
}

func main() {
    ctx, cancel := signal.NotifyContext(context.Background(), syscall.SIGINT, syscall.SIGTERM)
    defer cancel()

    // Conectar
    client := rmq.NewClient("amqp://admin:admin123@localhost:5672/")
    if err := client.Connect(ctx); err != nil {
        slog.Error("no se pudo conectar", "err", err)
        return
    }
    defer client.Close()

    // Producer
    producer, err := rmq.NewProducer(client)
    if err != nil { slog.Error("producer", "err", err); return }
    defer producer.Close()

    // Publicar un evento de prueba
    evento := OrdenCreada{OrdenID: "ord-001", UsuarioID: "usr-elkin", Total: 250_000}
    if err := producer.Publish(ctx, "ordenes.events", "orden.creada", evento); err != nil {
        slog.Error("publicar", "err", err)
    }
    slog.Info("evento publicado", "orden_id", evento.OrdenID)

    // Consumer con handler
    handler := func(ctx context.Context, msg amqp.Delivery) error {
        var orden OrdenCreada
        if err := json.Unmarshal(msg.Body, &orden); err != nil {
            return fmt.Errorf("deserializar: %w", err) // NACK → DLQ
        }
        slog.Info("procesando orden", "id", orden.OrdenID, "total", orden.Total)
        // lógica de negocio aquí...
        return nil
    }

    consumer := rmq.NewConsumer(client, "ordenes.creadas", "inventario-worker", handler)
    consumer.StartPool(ctx, 3) // 3 workers compitiendo

    <-ctx.Done()
    slog.Info("servicio detenido")
}

Delayed Message Exchange

El plugin rabbitmq_delayed_message_exchange añade un nuevo tipo de exchange (x-delayed-message) que retiene el mensaje en el broker durante N milisegundos antes de enrutarlo a la queue de destino. El delay se especifica por mensaje individualmente en el header x-delay. Esto simplifica enormemente implementar mensajes programados y retries con delay preciso, sin necesitar queues intermediarias con TTL ni schedulers externos.

        "x-delayed-type": "direct", // tipo subyacente
    },
)

    amqp.Publishing{
        Headers: amqp.Table{

        },
        ContentType:  "application/json",

    },

Clustering y Alta Disponibilidad

Cluster RabbitMQ

Un cluster de RabbitMQ es un grupo de nodos que trabajan juntos para garantizar alta disponibilidad. Si un nodo falla, los otros siguen atendiendo mensajes. La pieza clave son las Quorum Queues: replican los mensajes en múltiples nodos usando el algoritmo Raft, de forma que ningún mensaje se pierda aunque caiga un servidor.

go conectar con TLS
import (
    "crypto/tls"
    "crypto/x509"
    "os"
    amqp "github.com/rabbitmq/amqp091-go"
)

func connectTLS(host string) (*amqp.Connection, error) {
    // Cargar certificados
    cert, err := tls.LoadX509KeyPair("client.pem", "client-key.pem")
    if err != nil { return nil, err }

    caCert, err := os.ReadFile("ca.pem")
    if err != nil { return nil, err }

    pool := x509.NewCertPool()
    pool.AppendCertsFromPEM(caCert)

    tlsCfg := &tls.Config{
        Certificates: []tls.Certificate{cert},
        RootCAs:      pool,
        ServerName:   host,
        MinVersion:   tls.VersionTLS13,
    }

    return amqp.DialTLS(
        fmt.Sprintf("amqps://usuario:pass@%s:5671/vhost", host),
        tlsCfg,
    )
}
