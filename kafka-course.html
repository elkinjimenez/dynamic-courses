¿Qué es Kafka?

Historia, propósito y por qué domina el streaming de eventos

Apache Kafka es una plataforma distribuida de streaming de eventos creada originalmente en LinkedIn y donada a la Apache Software Foundation en 2011. Hoy es el estándar de facto para sistemas de mensajería de alto volumen y arquitecturas orientadas a eventos.

⚡

Alto Rendimiento

Millones de mensajes por segundo con latencias de milisegundos, gracias a escritura secuencial en disco.

📦

Durabilidad

Los mensajes se persisten en disco y se replican. No se pierden aunque un broker falle.

🔀

Escalabilidad

Escala horizontalmente añadiendo brokers y particiones sin downtime.

🔁

Replay

Los consumers pueden releer mensajes del pasado. No es un queue tradicional.

¿Por qué NO un sistema de mensajería tradicional?

Queue Tradicional (RabbitMQ)
  • Mensaje se borra al consumirse
  • Push model — el broker empuja
  • No replay de mensajes
  • Escala con dificultad
  • Ideal: tasks/jobs discretos
Kafka (Log distribuido)
  • Mensaje persiste (configurable)
  • Pull model — el consumer jala
  • Replay completo posible
  • Escala infinitamente
  • Ideal: streams, eventos, auditoría

Casos de uso reales

Empresa Uso Volumen
LinkedIn Tracking de actividad de usuarios, métricas operacionales 7 billones de mensajes/día
Uber Geolocalización en tiempo real, precios dinámicos 1 trillón de mensajes/año
Netflix Monitoreo, pipeline de datos, recomendaciones Billones de eventos/día
Banco de América Procesamiento de transacciones, detección de fraude Sub-segundo latencia

Arquitectura Core

Brokers, clusters, ZooKeeper/KRaft y el log distribuido

La arquitectura de Kafka se basa en un clúster de brokers distribuidos que almacenan y sirven streams de mensajes. Cada componente tiene un rol bien definido: los brokers persisten los datos, el Controller gestiona el estado del clúster, y los clients (producers y consumers) escriben y leen mensajes. Entender cómo interactúan estos piezas es esencial para diseñar sistemas robustos.

Componentes principales

🖥️

Broker

Un servidor Kafka. El cluster tiene 3+ brokers en producción. Cada broker tiene un ID único.

📚

Topic

Canal de mensajes nombrado. Similar a una tabla de base de datos o un directorio de archivos.

🗂️

Partition

División física de un topic. Cada partición es un log ordenado e inmutable.

👑

Controller

Broker especial que gestiona el estado del cluster: elecciones de líder, asignación de particiones.

📤

Producer

Aplicación que publica (escribe) mensajes a topics.

📥

Consumer

Aplicación que suscribe y lee mensajes de topics.

El Commit Log

El corazón de Kafka es el commit log distribuido. Cada partición es un archivo log append-only. Los mensajes se añaden al final y se identifican por su offset (posición en el log). Este diseño permite:

  • Escritura secuencial en disco — más rápida que escritura aleatoria incluso vs SSD
  • Lecturas O(1) por offset — acceso directo sin búsqueda
  • Replay — cualquier consumer puede volver al offset 0
Partition como Commit Log
offset: 0 1 2 3 4 ← nuevo ┌──────────┬──────────┬──────────┬──────────┬──────────┐ │ msg:pedido│msg:pago │msg:envio │msg:entrega│msg:review│ ← append only └──────────┴──────────┴──────────┴──────────┴──────────┘ ▲ ▲ Consumer A (offset 0) Consumer B (offset 3) — puede releer desde el inicio — solo lee los nuevos

ZooKeeper vs KRaft

⚠️
Kafka 3.x migró a KRaft (Kafka Raft). ZooKeeper fue deprecado en Kafka 2.8 y eliminado en Kafka 4.0. En producción moderna usa solo KRaft — no requiere ZooKeeper. Toda la metadata se gestiona internamente.
Modo Legacy (ZooKeeper)
  • Requiere proceso separado
  • Más complejo de operar
  • Kafka ≤ 2.7
  • Topología: Kafka + ZK cluster
KRaft Mode ✅ Recomendado
  • Sin dependencias externas
  • Arranque más rápido
  • Kafka ≥ 3.3 (GA)
  • Topología: solo Kafka cluster

⚡ Pregunta Rápida

A) Almacenar todos los mensajes del clúster en memoria para acceso rápido
B) Gestionar el estado del clúster: elecciones de líder de partición y metadatos de réplicas

Topics y Particiones

La unidad fundamental del diseño Kafka

El diseño correcto de topics y particiones es la decisión más importante en Kafka. Afecta rendimiento, escalabilidad, orden de mensajes y paralelismo.

Topics

Un topic es una categoría o feed de mensajes. Los productores publican en topics; los consumidores se suscriben a topics. Un topic puede tener múltiples particiones distribuidas en el cluster.

Convenciones de nombrado

nomenclatura
# Patrón recomendado: dominio.entidad.evento
pedidos.ordenes.creadas
pedidos.ordenes.actualizadas
pagos.transacciones.procesadas
usuarios.cuentas.registradas
inventario.productos.actualizados

# Patrones para microservicios
{servicio}.{recurso}.{accion}
{dominio}.{subdomain}.{evento}

# Comandos vs Eventos (importante para CQRS)
cmd.pedidos.crear-orden      # comando (imperativo)
evt.pedidos.orden-creada     # evento (pasado)

Particiones

Cada topic se divide en N particiones. Cada partición es un log independiente y ordenado. El número de particiones determina el paralelismo máximo de consumo.

Partición 0
off:0user:101
off:1user:201
off:2user:301
Partición 1
off:0user:102
off:1user:202
off:2user:302
Partición 2
off:0user:103
off:1user:203
off:2user:303

¿Cuántas particiones usar?

📐
Fórmula guía:
particiones = max(throughput_target / throughput_por_partition, max_consumers_paralelos)

Reglas prácticas:
  • Mínimo 1 partición por consumer en el grupo
  • En producción: empieza con 3–12 particiones
  • Topics de alta carga: 30–100+ particiones
  • Más particiones = más paralelismo pero más overhead de metadata

Partitioning Keys — Orden garantizado

El orden de mensajes en Kafka está garantizado dentro de una partición, NO a través del topic. La clave de partición determina a qué partición va cada mensaje.

concepto
# Sin clave → round-robin entre particiones → SIN orden global
Producer → P0: msg1, msg3
         → P1: msg2, msg4

# Con clave "order_id" → misma key = misma partición → ORDEN garantizado
Producer key:"order-123" → P2: msg1(order-123), msg2(order-123), msg3(order-123)
Producer key:"order-456" → P0: msg1(order-456), msg2(order-456)

# La key se hashea: partition = hash(key) % num_partitions
⚠️
Hot partition: Si todas tus claves apuntan a la misma partición (ej: clave es un enum con pocos valores), tendrás un desbalance. Diseña keys con alta cardinalidad: user_id, order_id, session_id.

Gestión de Topics — CLI

bash kafka-topics.sh
# Crear topic
kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --topic pedidos.ordenes.creadas \
  --partitions 6 \
  --replication-factor 3

# Listar topics
kafka-topics.sh --list --bootstrap-server localhost:9092

# Describir topic (particiones, líderes, réplicas)
kafka-topics.sh --describe \
  --topic pedidos.ordenes.creadas \
  --bootstrap-server localhost:9092

# Aumentar particiones (solo se puede aumentar, NUNCA decrementar)
kafka-topics.sh --alter \
  --topic pedidos.ordenes.creadas \
  --partitions 12 \
  --bootstrap-server localhost:9092

# Eliminar topic
kafka-topics.sh --delete \
  --topic pedidos.ordenes.creadas \
  --bootstrap-server localhost:9092
⚡ Pregunta Rápida
Si un producer envía mensajes con key="order-123", ¿qué garantiza Kafka sobre esos mensajes?
A) Los mensajes se distribuyen uniformemente entre todas las particiones del topic
B) Todos los mensajes con la misma key siempre van a la misma partición, garantizando orden entre ellos
C) Kafka garantiza orden global en todas las particiones del topic sin importar la key
✅ Correcto. La fórmula partición = hash(key) % num_particiones asegura que la misma key siempre aterrice en la misma partición. El orden se garantiza dentro de una partición, no a nivel del topic completo.

Producers

Publicar mensajes: configuraciones, garantías de entrega y batching

Un producer es cualquier aplicación que publica mensajes en topics de Kafka. Tiene configuraciones críticas que determinan el balance entre rendimiento, latencia y durabilidad.

Anatomía de un mensaje Kafka

Campo Tipo Descripción
key bytes (nullable) Determina la partición destino. Si null → round-robin.
value bytes El payload del mensaje. Puede ser JSON, Avro, Protobuf, etc.
headers KV pairs Metadata: correlationId, tracingId, content-type, etc.
timestamp int64 (ms) Tiempo de creación o ingesta. Usado para retención por tiempo.
partition int32 Partición de destino (calculada o explícita).
offset int64 Posición del mensaje en la partición (asignado por el broker).

Garantías de entrega (acks)

La configuración acks controla cuándo el broker confirma la escritura:

acks Semántica Rendimiento Riesgo de pérdida
0 Fire and forget — sin confirmación 🚀 Máximo Alta (si broker falla)
1 Ack del líder solamente ⚡ Alto Media (si líder falla antes de replicar)
all / -1 Ack de todas las réplicas sincronizadas (ISR) 🐢 Menor Mínima ✅ Recomendado para producción

Semánticas de entrega

Semántica Config Resultado Caso de uso
At-most-once acks=0, no retry Puede perder mensajes Logs no críticos, métricas
At-least-once acks=all, retries=MAX Duplicados posibles La mayoría de casos (idempotencia en consumer)
Exactly-once enable.idempotence=true, transacciones Sin pérdidas ni dups Pagos, transacciones financieras

Batching y rendimiento

🚀
Kafka agrupa mensajes en batches antes de enviarlos, aumentando dramáticamente el throughput. Los parámetros clave son batch.size (bytes por batch) y linger.ms (tiempo máximo de espera para llenar el batch).
properties producer.properties
# Configuración Producer de producción

# Conexión
bootstrap.servers=kafka1:9092,kafka2:9092,kafka3:9092

# Durabilidad ✅ recomendado
acks=all
enable.idempotence=true      # Producer idempotente (evita duplicados por retry)
retries=2147483647            # MAX_INT — reintentar indefinidamente
retry.backoff.ms=100

# Rendimiento
batch.size=65536              # 64KB por batch (default: 16KB)
linger.ms=5                   # esperar 5ms para llenar el batch
compression.type=snappy       # compresión: none, gzip, snappy, lz4, zstd
buffer.memory=33554432        # 32MB de buffer en memoria

# Timeouts
request.timeout.ms=30000
delivery.timeout.ms=120000

# Serialización
key.serializer=org.apache.kafka.common.serialization.StringSerializer
value.serializer=org.apache.kafka.common.serialization.ByteArraySerializer

Diagrama de flujo del Producer

App ─▶ send(record) ─▶ Serializer ─▶ Partitioner ─▶ RecordAccumulator (hash(key)%n) (buffer de batches) │ linger.ms o batch lleno ▼ Sender Thread ─▶ Broker ─▶ ack ─▶ callback
⚡ Pregunta Rápida
¿Qué configuración de acks es la recomendada en producción para garantizar durabilidad sin perder mensajes?
A) acks=0 — fire and forget, máximo rendimiento
B) acks=1 — solo el broker líder confirma la escritura
C) acks=all con enable.idempotence=true — confirmación de todas las réplicas sincronizadas (ISR)
✅ Correcto. acks=all espera confirmación de todas las réplicas en el ISR. Combinado con enable.idempotence=true, evita tanto la pérdida de mensajes como los duplicados por reintentos.

Consumers y Consumer Groups

Leer mensajes, grupos de consumo y rebalanceo

Los consumers leen mensajes de Kafka de forma activa (pull model). La abstracción de Consumer Group permite escalar el consumo horizontalmente y garantizar que cada mensaje sea procesado por exactamente un consumer dentro del grupo.

Consumer Groups

Topic: pedidos (3 particiones) P0 ──▶ Consumer-1 ┐ P1 ──▶ Consumer-2 ├── Group: procesamiento-pedidos P2 ──▶ Consumer-3 ┘ Mismo topic, otro grupo independiente: P0 ──▶ Consumer-A ┐ P1 ──▶ Consumer-B ├── Group: analytics-pedidos P2 ──▶ Consumer-B ┘ (2 consumers → 1 cubre 2 particiones) Regla: una partición → máximo 1 consumer activo por grupo si consumers > particiones → algunos consumers idle

Offset Management

El offset es la posición del próximo mensaje a leer. Los consumers lo gestionan explícitamente. Kafka guarda los offsets commiteados en el topic interno __consumer_offsets.

Configuración Comportamiento Cuándo usar
enable.auto.commit=true Kafka hace commit automáticamente cada auto.commit.interval.ms Casos simples, puede perder mensajes si el proceso falla antes del commit
enable.auto.commit=false El consumer hace commit manualmente después de procesar ✅ Producción — control total, at-least-once garantizado

Configuraciones clave del Consumer

properties consumer.properties
# Conexión y grupo
bootstrap.servers=kafka1:9092,kafka2:9092,kafka3:9092
group.id=mi-servicio-consumer       # identificador del grupo
client.id=mi-servicio-instancia-1

# Offset inicial (cuando no hay offset previo)
auto.offset.reset=earliest   # leer desde el principio
# auto.offset.reset=latest   # solo mensajes nuevos

# Commit manual (recomendado en producción)
enable.auto.commit=false

# Polling y heartbeat
max.poll.records=500              # máximo mensajes por poll()
max.poll.interval.ms=300000       # 5 min — tiempo máximo entre polls
session.timeout.ms=45000          # timeout para detectar consumer muerto
heartbeat.interval.ms=3000        # intervalo de heartbeat

# Deserialización
key.deserializer=...StringDeserializer
value.deserializer=...ByteArrayDeserializer

Rebalanceo

El rebalanceo ocurre cuando cambia la composición del grupo (consumer entra/sale/falla). Durante el rebalanceo, todos los consumers detienen el consumo (stop-the-world). Estrategias:

Estrategia Comportamiento Recomendación
Range (default) Asigna rangos contiguos de particiones Desbalanceo en tópicos con muchas particiones
RoundRobin Distribuye uniformemente entre todos los consumers Mejor distribución, más movimientos
Sticky Mínimos movimientos de particiones al rebalancear ✅ Mejor rendimiento general
CooperativeSticky Rebalanceo incremental, sin stop-the-world ✅✅ Mejor para producción Kafka 2.4+
⚡ Pregunta Rápida
Tienes un topic con 6 particiones y un Consumer Group con 8 consumers. ¿Qué ocurre?
A) Error — no puede haber más consumers que particiones
B) 6 consumers activos, 2 en standby (idle)
C) Kafka reparte las 6 particiones entre los 8 consumers fraccionadamente
✅ Correcto. Una partición solo puede ser asignada a UN consumer por grupo. Los consumers extra quedan en standby y tomarán el relevo si algún consumer activo falla.

Replicación y Durabilidad

ISR, factores de replicación y tolerancia a fallos

La replicación es el mecanismo que hace a Kafka tolerante a fallos. Cada partición tiene un broker líder y N réplicas (followers) en otros brokers.

In-Sync Replicas (ISR)

Partition 0 del topic "pedidos" — RF=3 Broker-1 (Líder P0) ◄── escrituras y lecturas │ log: [0,1,2,3,4] ← offset latest = 4 │ ├──replicate──▶ Broker-2 (Follower P0) │ log: [0,1,2,3,4] ✅ ISR │ └──replicate──▶ Broker-3 (Follower P0) log: [0,1,2,3] ⚠️ fuera del ISR (lag=1) ISR = {Broker-1, Broker-2} → acks=all requiere ack de AMBOS Si Broker-1 falla → Broker-2 se convierte en nuevo líder

Configuraciones de replicación

bash configuración crítica
# Al crear el topic
kafka-topics.sh --create \
  --topic pedidos.ordenes.creadas \
  --partitions 6 \
  --replication-factor 3 \    # 3 copias — RF mínimo en producción
  --bootstrap-server localhost:9092

# Configuraciones del broker (server.properties)
default.replication.factor=3
min.insync.replicas=2          # mínimo 2 réplicas sincronizadas para aceptar writes
                               # con RF=3, sobrevive fallo de 1 broker
unclean.leader.election.enable=false  # evitar pérdida de datos
                                       # (no elegir followers desactualizados como líder)
🛡️
Combinación de oro para producción:
replication-factor=3 + min.insync.replicas=2 + acks=all
→ Tolera fallo de 1 broker sin pérdida de datos y sin interrumpir escrituras.

Compactación de logs

Además de la retención por tiempo/tamaño, Kafka tiene un modo especial: log compaction. Retiene solo el último valor para cada key, eliminando versiones anteriores.

Antes de compactación: key:user-1 → valor:"Juan" offset:0 key:user-2 → valor:"Ana" offset:1 key:user-1 → valor:"Juan C" offset:2 ← nuevo valor para user-1 key:user-3 → valor:"Pedro" offset:3 key:user-2 → valor:null offset:4 ← tombstone = eliminar user-2 Después de compactación: key:user-1 → valor:"Juan C" offset:2 ← solo el último key:user-3 → valor:"Pedro" offset:3 ← se mantiene key:user-2 eliminado (tombstone)
💡
Log compaction es ideal para: tablas de estado (state tables), cachés distribuidos, Event Sourcing donde solo necesitas el estado actual por entidad.
⚡ Pregunta Rápida
Con replication-factor=3 y min.insync.replicas=2, ¿cuántos brokers pueden fallar sin perder datos ni interrumpir escrituras?
A) 0 brokers — cualquier fallo detiene el cluster
B) 1 broker — quedan 2 réplicas activas, suficiente para el mínimo de ISR
C) 2 brokers — el factor de replicación 3 permite 2 fallos simultáneos
✅ Correcto. Con RF=3 y min.ISR=2, necesitas mínimo 2 réplicas vivas para que el cluster acepte escrituras. Si cae 1 broker aún quedan 2 réplicas (cumple min.ISR). Si caen 2 brokers, solo queda 1 réplica — el cluster rechaza escrituras para evitar pérdida de datos.

Offsets y Retención

Posicionamiento, replay y políticas de limpieza

El offset es el identificador numérico secuencial de cada mensaje dentro de una partición. Controlar dónde comienza a leer un consumer — y cuánto tiempo Kafka conserva los mensajes — es clave para implementar replay de eventos, tolerancia a fallos y gestión eficiente del almacenamiento.

Gestión manual de Offsets

bash kafka-consumer-groups.sh
# Ver estado de un consumer group (lag, offsets)
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group mi-servicio-consumer \
  --describe

# Salida:
# GROUP          TOPIC            PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG   CONSUMER-ID
# mi-servicio    pedidos.ordenes  0          145             150             5     host-1
# mi-servicio    pedidos.ordenes  1          200             200             0     host-2
# mi-servicio    pedidos.ordenes  2          98              110             12    host-1

# Resetear offset — REPLAY de mensajes
# El consumer group DEBE estar detenido

# Desde el principio (todos los mensajes)
kafka-consumer-groups.sh --reset-offsets \
  --group mi-servicio-consumer \
  --topic pedidos.ordenes.creadas \
  --to-earliest --execute \
  --bootstrap-server localhost:9092

# Desde un punto en el tiempo
kafka-consumer-groups.sh --reset-offsets \
  --group mi-servicio-consumer \
  --topic pedidos.ordenes.creadas \
  --to-datetime 2025-01-15T00:00:00.000 --execute \
  --bootstrap-server localhost:9092

# Mover N mensajes atrás
kafka-consumer-groups.sh --reset-offsets \
  --group mi-servicio-consumer \
  --topic pedidos.ordenes.creadas \
  --shift-by -1000 --execute \
  --bootstrap-server localhost:9092

Políticas de Retención

Configuración Default Descripción
retention.ms 7 días Tiempo que se retienen los mensajes. -1 = infinito.
retention.bytes -1 (inf) Tamaño máximo del log por partición.
cleanup.policy delete delete elimina viejos, compact mantiene último por key, compact,delete ambos.
segment.bytes 1GB Tamaño de cada segmento de log (archivo físico).
segment.ms 7 días Kafka rota el segmento activo aunque no esté lleno.
bash topic retention config
# Configurar retención al crear
kafka-topics.sh --create \
  --topic auditoria.eventos \
  --partitions 6 \
  --replication-factor 3 \
  --config retention.ms=604800000 \    # 7 días
  --config cleanup.policy=compact \   # solo último valor por key
  --bootstrap-server localhost:9092

# Modificar retención en caliente
kafka-configs.sh --alter \
  --entity-type topics \
  --entity-name auditoria.eventos \
  --add-config retention.ms=2592000000 \  # 30 días
  --bootstrap-server localhost:9092

Consumer Lag

El consumer lag es la diferencia entre el último offset disponible y el último offset procesado por el consumer. Es la métrica más crítica para monitorear la salud de un consumer.

Log-End-Offset (LEO): 1000 ← último mensaje en el broker Current Offset: 985 ← último procesado por consumer ───── LAG: 15 ← mensajes pendientes de procesar Lag = 0 → consumer al día ✅ Lag creciente → consumer lento, escalar ⚠️ Lag > umbral → alertar 🚨
⚡ Pregunta Rápida
¿Qué condición es obligatoria para resetear los offsets de un consumer group con --reset-offsets --execute?
A) No hay ninguna condición, se puede hacer en cualquier momento con el grupo activo
B) El consumer group debe estar completamente detenido (sin consumers activos corriendo)
C) El topic debe estar en modo compactación para permitir el reset
✅ Correcto. Kafka no permite resetear offsets de un grupo con consumers activos. Si hay instancias corriendo, el reset falla o entra en conflicto con los commits en curso. Siempre detiene todos los consumers del grupo antes de ejecutar el reset.

Serialización y Schema Registry

JSON, Avro, Protobuf y contratos de datos

Kafka transporta bytes crudos. La serialización/deserialización es responsabilidad del producer y consumer. Elegir el formato correcto impacta rendimiento, compatibilidad y evolución del schema.

Schema Registry

Confluent Schema Registry es un servicio central que almacena y versiona schemas. Permite evolución de schemas sin romper consumers existentes.

Publica a Kafka Consumer: 1. Lee bytes del topic 2. Extrae schema_id=5 3. Consulta Schema Registry → obtiene schema V1 4. Deserializa usando ese schema Schema Registry cachea schemas localmente → solo 1 req HTTP por schema_id

}

// Deserializar al consumir
func Deserialize(data []byte) (*OrdenCreadaEvent, error) {
    var event OrdenCreadaEvent
    if err := json.Unmarshal(data, &event); err != nil {
        return nil, fmt.Errorf("deserializar evento: %w", err)
    }
    return &event, nil
}

type EventEnvelope struct {
    EventID   string          `json:"event_id"`
    EventType string          `json:"event_type"`
    Version   int             `json:"version"`
    Timestamp time.Time       `json:"timestamp"`
    Payload   json.RawMessage `json:"payload"`  // raw para deserializar según tipo
}

func DispatchEvent(data []byte, handlers map[string]EventHandler) error {
    var env EventEnvelope
    if err := json.Unmarshal(data, &env); err != nil {
        return err
    }
    handler, ok := handlers[env.EventType]
    if !ok {
        return fmt.Errorf("handler no encontrado para: %s", env.EventType)
    }
    return handler.Handle(env)
}

type Producer struct {
    client *kgo.Client
    logger *slog.Logger
}

type ProducerConfig struct {
    Brokers          []string
    Topic            string
    MaxRetries       int
    RequiredAcks     kgo.Acks
    CompressionCodec kgo.CompressionCodec
}

func DefaultProducerConfig(brokers []string) ProducerConfig {
    return ProducerConfig{
        Brokers:          brokers,
        MaxRetries:       10,
        RequiredAcks:     kgo.AllISRAcks(),       // acks=all
        CompressionCodec: kgo.SnappyCompression(), // snappy
    }
}

func NewProducer(cfg ProducerConfig) (*Producer, error) {
    opts := []kgo.Opt{
        kgo.SeedBrokers(cfg.Brokers...),
        kgo.RequiredAcks(cfg.RequiredAcks),
        kgo.WithLogger(kgo.BasicLogger(nil, kgo.LogLevelInfo, nil)),

        // Idempotencia (acks=all la activa automáticamente en franz-go)
        kgo.ProducerBatchMaxBytes(1_000_000),   // 1MB por batch
        kgo.BrokerMaxWriteBytes(100_000_000),

        // Retries
        kgo.RecordRetries(cfg.MaxRetries),
        kgo.RetryBackoffFn(func(try int) time.Duration {
            return time.Duration(try) * 100 * time.Millisecond
        }),
    }

    client, err := kgo.NewClient(opts...)
    if err != nil {
        return nil, fmt.Errorf("crear producer kafka: %w", err)
    }

    // Verificar conectividad
    if err = client.Ping(context.Background()); err != nil {
        return nil, fmt.Errorf("kafka ping fallido: %w", err)
    }

    return &Producer{client: client, logger: slog.Default()}, nil
}

// Publicar un evento serializado como JSON
func (p *Producer) Publish(ctx context.Context, topic, key string, payload any) error {
    data, err := json.Marshal(payload)
    if err != nil {
        return fmt.Errorf("serializar payload: %w", err)
    }

    record := &kgo.Record{
        Topic: topic,
        Key:   []byte(key),
        Value: data,
        Headers: []kgo.RecordHeader{
            {Key: "content-type", Value: []byte("application/json")},
            {Key: "producer",     Value: []byte("pedidos-service")},
        },
    }

    // ProduceSync — espera confirmación (usa ProduceAsync para fire-and-forget)
    results := p.client.ProduceSync(ctx, record)
    if err := results.FirstErr(); err != nil {
        return fmt.Errorf("publicar a %s: %w", topic, err)
    }

    p.logger.InfoContext(ctx, "mensaje publicado",
        "topic", topic,
        "key", key,
        "partition", results[0].Record.Partition,
        "offset", results[0].Record.Offset,
    )
    return nil
}

// PublishBatch — publicar múltiples mensajes eficientemente
func (p *Producer) PublishBatch(ctx context.Context, records []*kgo.Record) error {
    results := p.client.ProduceSync(ctx, records...)
    return results.FirstErr()
}

func (p *Producer) Close() {
    p.client.Close()
}

package kafka

import (
    "context"
    "encoding/json"
    "fmt"
    "log/slog"

    "github.com/twmb/franz-go/pkg/kgo"
)

// MessageHandler procesa un mensaje — implementar en cada servicio
type MessageHandler func(ctx context.Context, record *kgo.Record) error

type Consumer struct {
    client  *kgo.Client
    handler MessageHandler
    logger  *slog.Logger
}

type ConsumerConfig struct {
    Brokers         []string
    GroupID         string
    Topics          []string
    AutoOffsetReset string  // "earliest" | "latest"
    MaxPollRecords  int
}

func NewConsumer(cfg ConsumerConfig, handler MessageHandler) (*Consumer, error) {
    var offsetStart kgo.Offset
    if cfg.AutoOffsetReset == "earliest" {
        offsetStart = kgo.NewOffset().AtStart()
    } else {
        offsetStart = kgo.NewOffset().AtEnd()
    }

    opts := []kgo.Opt{
        kgo.SeedBrokers(cfg.Brokers...),
        kgo.ConsumerGroup(cfg.GroupID),
        kgo.ConsumeTopics(cfg.Topics...),
        kgo.ConsumeResetOffset(offsetStart),

        // CooperativeSticky — rebalanceo incremental sin stop-the-world
        kgo.Balancers(kgo.CooperativeStickyBalancer()),

        // Auto commit cada 5s (o usa CommitMarkedOffsets para control manual)
        kgo.AutoCommitMarks(),
        kgo.BlockRebalanceOnPoll(),
    }

    client, err := kgo.NewClient(opts...)
    if err != nil {
        return nil, fmt.Errorf("crear consumer kafka: %w", err)
    }

    return &Consumer{
        client:  client,
        handler: handler,
        logger:  slog.Default(),
    }, nil
}

// Start inicia el loop de consumo — bloquea hasta que ctx se cancele
func (c *Consumer) Start(ctx context.Context) error {
    c.logger.Info("consumer iniciado", "topics", c.client.GetConsumeTopics())

    for {
        // Fetch bloquea hasta tener mensajes o que ctx se cancele
        fetches := c.client.PollFetches(ctx)

        if ctx.Err() != nil {
            c.logger.Info("consumer detenido por contexto")
            return nil
        }

        if errs := fetches.Errors(); len(errs) > 0 {
            for _, e := range errs {
                c.logger.Error("error en fetch", "err", e.Err,
                    "topic", e.Topic, "partition", e.Partition)
            }
            continue
        }

        // Procesar mensajes del batch
        fetches.EachRecord(func(record *kgo.Record) {
            if err := c.processRecord(ctx, record); err != nil {
                c.logger.Error("error procesando mensaje",
                    "err", err,
                    "topic", record.Topic,
                    "partition", record.Partition,
                    "offset", record.Offset,
                )
                // En producción: enviar a DLQ (Dead Letter Queue)
                return
            }
            // Marcar como procesado para commit
            c.client.MarkCommitRecords(record)
        })
    }
}

func (c *Consumer) processRecord(ctx context.Context, r *kgo.Record) error {
    c.logger.DebugContext(ctx, "procesando mensaje",
        "topic", r.Topic,
        "partition", r.Partition,
        "offset", r.Offset,
        "key", string(r.Key),
    )
    return c.handler(ctx, r)
}

func (c *Consumer) Close() {
    c.client.Close()
}

Kafka Streams

Procesamiento stateful de streams en tiempo real

Kafka Streams es una librería de procesamiento de streams (no un cluster separado). Permite transformar, agregar y enriquecer streams de datos en tiempo real, con estado distribuido.

Conceptos clave

📨

KStream

Stream de registros infinito. Cada registro es independiente — insert semantics. Ej: log de clicks, transacciones.

📋

KTable

Vista materializada de un stream. Solo el último valor por key — update semantics. Ej: estado de usuario.

🗂️

State Store

Almacenamiento local (RocksDB) para operaciones stateful: aggregations, joins, windowing.

🪟

Windows

Agrupar eventos por tiempo: Tumbling (fijo), Hopping (solapado), Session (por actividad).

Topología de procesamiento

Pipeline: detectar fraude en tiempo real KStream: pagos.transacciones │ ├─ filter(monto > 0) │ ├─ mapValues(enriquecer con datos de usuario) │ ├─ groupByKey(usuario_id) │ ├─ windowedBy(TumblingWindow(5 minutos)) │ ├─ aggregate(sumar montos) │ ├─ filter(total_5min > 10_000_000) ← umbral sospechoso │ └─▶ KStream: fraude.alertas ──▶ Servicio de alertas

Windowing

Window Type Descripción Caso de uso
Tumbling Ventanas fijas sin solapamiento. Ej: cada 5 minutos. Reportes periódicos, métricas por intervalo
Hopping Ventanas fijas con solapamiento. Ej: 5min cada 1min. Sliding averages, detección de anomalías
Session Ventanas dinámicas basadas en inactividad de un key. Sesiones de usuario, análisis de comportamiento
Sliding Se actualiza con cada evento. Siempre el mismo tamaño de tiempo. Alertas inmediatas, rate limiting

type PipelineProcessor struct {
    consumer *kafka.Consumer
    producer *kafka.Producer
    enrich   func(ctx context.Context, pago *Pago) (*PagoEnriquecido, error)
}

func (p *PipelineProcessor) Process(ctx context.Context, record *kgo.Record) error {
    // 1. Deserializar
    var pago Pago
    if err := json.Unmarshal(record.Value, &pago); err != nil {
        return err
    }

    // 2. Transformar / enriquecer
    enriquecido, err := p.enrich(ctx, &pago)
    if err != nil {
        return fmt.Errorf("enriquecer pago: %w", err)
    }

    // 3. Publicar al topic de salida
    return p.producer.Publish(
        ctx,
        "pagos.enriquecidos",
        pago.PagoID,
        enriquecido,
    )
}

// Aggregation en memoria (para volúmenes moderados)
type WindowAggregator struct {
    mu      sync.RWMutex
    windows map[string]*windowState  // key → estado
    windowDuration time.Duration
}

func (w *WindowAggregator) Add(key string, monto float64) {
    w.mu.Lock()
    defer w.mu.Unlock()

    now := time.Now()
    state := w.windows[key]
    if state == nil || now.Sub(state.StartTime) > w.windowDuration {
        state = &windowState{StartTime: now}
        w.windows[key] = state
    }
    state.Total += monto
    state.Count++
}

func (w *WindowAggregator) GetTotal(key string) float64 {
    w.mu.RLock()
    defer w.mu.RUnlock()
    if s := w.windows[key]; s != nil {
        return s.Total
    }
    return 0
}

B) KStream es un flujo de eventos inmutables (insert semantics); KTable representa el último estado por key (update semantics, solo conserva el valor más reciente)
C) Son equivalentes, solo cambia la sintaxis de la API para diferentes tipos de operaciones
✅ Correcto. Un KStream modela eventos discretos (cada registro es independiente). Una KTable modela un estado: si llega un nuevo registro con la misma key, actualiza el valor anterior. La KTable es como una vista materializada en tiempo real del último estado por entidad.

Patrones de Diseño

Event Sourcing, CQRS, Saga Pattern y Dead Letter Queue

Los sistemas basados en Kafka adoptan patrones de arquitectura que van más allá de la mensajería simple. Event Sourcing usa Kafka como la única fuente de verdad del sistema. CQRS separa escrituras de lecturas para escalar independientemente. El Saga Pattern coordina transacciones distribuidas entre microservicios. La Dead Letter Queue captura mensajes fallidos para análisis y reprocesamiento.

Event Sourcing

En lugar de guardar el estado actual en BD, guardas la secuencia de eventos que llevaron a ese estado. Kafka es el log de eventos. El estado se reconstruye reproduciendo los eventos.

Base de datos tradicional: tabla ordenes: { id:1, estado:"entregada", total:50000 } ← estado actual Event Sourcing con Kafka: Kafka topic "ordenes": offset:0 { "evento":"orden-creada", "orden_id":1, "total":50000 } offset:1 { "evento":"pago-recibido", "orden_id":1 } offset:2 { "evento":"orden-enviada", "orden_id":1, "tracking":"ABC" } offset:3 { "evento":"orden-entregada","orden_id":1 } Estado = reducir(todos los eventos de esa orden)

CQRS (Command Query Responsibility Segregation)

CQRS + Kafka Command Side: REST POST /ordenes ──▶ validate ──▶ publish(orden-creada) ──▶ Kafka Query Side: Kafka ──▶ consumer ──▶ materializar en PostgreSQL/Redis (vista optimizada para consultas) REST GET /ordenes/:id ──▶ leer de PostgreSQL/Redis Ventaja: escrituras y lecturas escalan independientemente Las consultas no bloquean las escrituras

Saga Pattern — Transacciones distribuidas

El Saga Pattern coordina transacciones que abarcan múltiples microservicios usando eventos de compensación en caso de fallo.

Choreography-based Saga: Flujo de Pedido [Pedidos] ──evt:orden-creada──▶ [Inventario] │ ┌─────────────────────────┤ ▼ ▼ OK: evt:inventario-reservado FAIL: evt:inventario-sin-stock │ │ ▼ ▼ [Pagos] [Pedidos] cancelar-orden │ OK: evt:pago-procesado │ ▼ [Envíos] ──▶ evt:envio-programado

Dead Letter Queue (DLQ)

Cuando un consumer no puede procesar un mensaje (error de deserialización, lógica de negocio, servicio externo caído), se envía al DLQ para análisis y reprocesamiento posterior.

go dead letter queue
type DLQHandler struct {
    mainHandler MessageHandler
    dlqProducer *kafka.Producer
    maxRetries  int
}

type FailedMessage struct {
    OriginalTopic     string    `json:"original_topic"`
    OriginalPartition int32     `json:"original_partition"`
    OriginalOffset    int64     `json:"original_offset"`
    OriginalKey       string    `json:"original_key"`
    OriginalValue     string    `json:"original_value"`
    ErrorMessage      string    `json:"error_message"`
    RetryCount        int       `json:"retry_count"`
    FailedAt          time.Time `json:"failed_at"`
}

func (d *DLQHandler) Handle(ctx context.Context, record *kgo.Record) error {
    var lastErr error

    for attempt := 0; attempt <= d.maxRetries; attempt++ {
        if attempt > 0 {
            time.Sleep(time.Duration(attempt) * 500 * time.Millisecond)
        }

        if err := d.mainHandler(ctx, record); err != nil {
            lastErr = err
            slog.WarnContext(ctx, "reintento",
                "attempt", attempt, "err", err,
                "topic", record.Topic, "offset", record.Offset)
            continue
        }
        return nil // éxito
    }

    // Enviar al DLQ después de agotar reintentos
    dlqMsg := FailedMessage{
        OriginalTopic:     record.Topic,
        OriginalPartition: record.Partition,
        OriginalOffset:    record.Offset,
        OriginalKey:       string(record.Key),
        OriginalValue:     string(record.Value),
        ErrorMessage:      lastErr.Error(),
        RetryCount:        d.maxRetries,
        FailedAt:          time.Now(),
    }

    dlqTopic := record.Topic + ".dlq"
    if err := d.dlqProducer.Publish(ctx, dlqTopic, string(record.Key), dlqMsg); err != nil {
        return fmt.Errorf("enviar a DLQ: %w", err)
    }

    slog.ErrorContext(ctx, "mensaje enviado a DLQ",
        "dlq_topic", dlqTopic,
        "original_error", lastErr)
    return nil // retornar nil para no bloquear el consumer
}
⚡ Pregunta Rápida
En el Saga Pattern tipo Choreography, ¿cómo se coordinan los microservicios ante un fallo?
A) Un orquestador central detecta el fallo y llama a los servicios de compensación en orden
B) El servicio que falla publica un evento de fallo; otros servicios reaccionan a ese evento publicando sus propios eventos de compensación
C) Se usa una base de datos compartida y transacciones ACID distribuidas para revertir los cambios
✅ Correcto. En Choreography-Saga no hay orquestador. Cada servicio conoce solo su próximo paso. Ante un fallo, publica un evento de compensación (ej. inventario-sin-stock) y los servicios afectados reaccionan ejecutando su reversión.

Seguridad en Kafka

TLS, SASL, ACLs y mejores prácticas

Un clúster Kafka en producción expuesto sin seguridad es una vulnerabilidad crítica. Kafka implementa seguridad en tres capas complementarias: TLS cifra los datos en tránsito, SASL autentica la identidad de cada cliente, y las ACLs controlan qué operaciones puede ejecutar cada usuario en cada topic o consumer group.

Capas de seguridad

🔐

Cifrado (TLS)

Encripta datos en tránsito entre clients y brokers. Evita eavesdropping en la red.

🪪

Autenticación (SASL)

Verifica la identidad del cliente. Opciones: PLAIN, SCRAM-SHA-512, GSSAPI (Kerberos), OAUTHBEARER.

🛡️

Autorización (ACLs)

Controla qué operaciones puede hacer cada cliente en cada topic/grupo.

🔍

Auditoría

Registro de accesos y operaciones para compliance y detección de anomalías.

go lag monitor con franz-go
// Monitorear consumer lag programáticamente
func monitorLag(ctx context.Context, client *kgo.Client,
    group, topic string, alertThreshold int64) {

    ticker := time.NewTicker(30 * time.Second)
    defer ticker.Stop()

    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            // Obtener offsets del grupo
            offsets, err := client.FetchOffsetsForTopicPartitions(ctx,
                kgo.GroupOffsets(group),
                map[string][]int32{topic: nil},
            )
            if err != nil {
                slog.Error("fetch offsets", "err", err)
                continue
            }

            // Calcular lag por partición
            for partition, offset := range offsets[topic] {
                lag := offset.At.Lag // franz-go calcula el lag
                slog.Info("consumer lag",
                    "group", group,
                    "topic", topic,
                    "partition", partition,
                    "lag", lag,
                )
                if lag > alertThreshold {
                    slog.Warn("⚠️  LAG ALTO — considerar escalar consumers",
                        "partition", partition, "lag", lag,
                        "threshold", alertThreshold)
                }
            }
        }
    }
}
